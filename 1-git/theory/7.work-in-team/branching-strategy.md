# Стратегии ветвления (Git Flow, GitHub Flow, Trunk-Based)

## 📌 Зачем нужны стратегии ветвления

Когда над проектом работает команда, без правил начинается хаос:

- разные фичи ломают друг друга
- сложно делать релизы
- возникают конфликты
- непонятно, где “стабильный код”

👉 **Стратегия ветвления = набор правил, как жить в Git**

---

# 🌊 Git Flow (сложный, структурированный)

## 🧠 Идея
Чёткая система для проектов с релизами по расписанию.

---

## 🌿 Основные ветки

- `main` → только стабильный production-код
- `develop` → интеграция новых фич

```text
main ← стабильная версия
develop ← активная разработка
🔧 Вспомогательные ветки
Feature
git checkout -b feature/login develop

👉 новые функции

Release
git checkout -b release/1.2.0 develop

👉 подготовка к релизу

Hotfix
git checkout -b hotfix/1.2.1 main

👉 срочные исправления production

📦 Как выглядит поток
feature → develop → release → main
                    ↘ hotfix ↗
✅ Плюсы Git Flow
строгая структура
удобно для крупных проектов
поддержка нескольких версий
❌ Минусы
сложный процесс
много веток
медленные релизы
🚀 GitHub Flow (простой и популярный)
🧠 Идея

Одна главная ветка + короткоживущие feature-ветки.

🌿 Правила
main всегда готов к deploy
создаём ветку → PR → review → merge
🔧 Workflow
git checkout -b add-search
git push origin add-search

👉 дальше Pull Request

📦 Поток
feature branch → PR → main → deploy
⚡ Особенности
нет develop ветки
нет сложной иерархии
быстрый CI/CD
✅ Плюсы
простота
быстрые релизы
идеально для web/SaaS
❌ Минусы
сложнее работать с несколькими версиями
требует хорошего CI/CD
🌲 Trunk-Based Development (максимально простой)
🧠 Идея

Все изменения почти сразу попадают в main (trunk).

🌿 Основной принцип

👉 ветки живут очень недолго (часа/дни)

🎛 Feature flags вместо веток
if (flags.newSearch) {
  return newSearch();
} else {
  return oldSearch();
}

👉 код в production есть, но выключен

📦 Поток
main ← всё развитие идёт сюда напрямую
⚡ Требования
сильные тесты
быстрый CI/CD
дисциплина команды
feature flags
✅ Плюсы
максимальная скорость
минимум merge конфликтов
постоянная интеграция
❌ Минусы
сложно для junior команд
нужна сильная инфраструктура
сложность feature flags
📊 Сравнение стратегий
Стратегия	Сложность	Скорость	Где используется
Git Flow	высокая	медленная	enterprise, mobile
GitHub Flow	средняя	быстрая	web, SaaS
Trunk-Based	низкая (для системы), высокая (для команды)	очень быстрая	big tech
🧭 Когда что использовать
Git Flow
релизы раз в недели/месяцы
несколько версий в поддержке
GitHub Flow
web-приложения
частые деплои
небольшие команды
Trunk-Based
высоконагруженные системы
CI/CD-first компании
максимальная скорость разработки
🔄 Важная идея

👉 Нет “лучшей” стратегии

Команды часто:

упрощают Git Flow → GitHub Flow
добавляют release ветки
комбинируют подходы
💡 Главное
стратегия = правила командной работы
важно не название, а дисциплина
CI/CD делает современные модели возможными

👉 Хороший workflow = быстрые релизы без хаоса
