# Теги и управление релизами

## 📌 Что такое теги

**Тег (tag)** — это именованная ссылка на конкретный коммит.

- В отличие от веток — **не двигается**
- Используется как **фиксированная точка в истории**
- Чаще всего — для обозначения релизов

Примеры:
- `v1.0.0`
- `v2.1.3`
- `release-2023-10-15`

👉 Тег всегда указывает на один и тот же коммит — это гарантирует воспроизводимость версии.

---

## 🏷️ Типы тегов

### 1. Lightweight (легковесные)

Просто указатель на коммит без метаданных.

```bash
git tag v1.0.0
git tag v0.9.0 abc123def

Когда использовать:

локальные метки
временные состояния
2. Annotated (аннотированные)

Полноценный объект с информацией:

автор
дата
описание
(опционально) подпись
git tag -a v1.0.0 -m "First stable release"

Когда использовать:

релизы (рекомендуется)
важные версии
🔢 Семантическое версионирование (SemVer)

Формат:

MAJOR.MINOR.PATCH
Правила:
MAJOR (2.0.0) — несовместимые изменения
MINOR (1.2.0) — новая функциональность (совместимая)
PATCH (1.2.1) — исправления багов
Примеры:
v1.0.0 — первый релиз
v1.0.1 — багфикс
v1.1.0 — новая функция
v2.0.0 — breaking changes
🔍 Работа с тегами
Просмотр
git tag                # список
git tag -l "v1.*"      # фильтр
git show v1.0.0        # детали
Где вы относительно тега
git describe --tags

Пример:

v1.2.0-14-g2fc4d8a
🚀 Отправка тегов

⚠️ Теги не пушатся автоматически

git push origin v1.0.0        # один тег
git push origin --tags        # все
git push origin --follow-tags # только аннотированные (лучше)
❌ Удаление тегов
git tag -d v1.0.0                     # локально
git push origin --delete v1.0.0       # на сервере

⚠️ Не удаляйте опубликованные релизы — это ломает зависимые системы.

📦 Стратегии релизов
1. Релиз из main
git checkout main
git pull
git tag -a v1.2.0 -m "Release"
git push origin v1.2.0

✔ Просто
✔ Подходит для CI/CD

2. Release branch
git checkout -b release/1.2.0 main
# фиксы
git tag -a v1.2.0

✔ Стабилизация версии
✔ Параллельная разработка

3. Hotfix
git checkout -b hotfix/1.2.1 v1.2.0
# фикс
git tag -a v1.2.1

✔ Быстрое исправление продакшена

🤖 Автоматизация релизов
Генерация changelog
git log v1.1.0..v1.2.0 --pretty=format:"- %s"

Фильтрация:

--grep="^feat:"
--grep="^fix:"
CI/CD (по тегам)
on:
  push:
    tags:
      - 'v*'

👉 При создании тега автоматически запускается сборка/релиз

👥 Работа в команде

Важно договориться:

кто создаёт релизы (обычно lead)
единый формат версий
процесс релиза (чеклист)

📌 Рекомендуется:

использовать annotated теги
придерживаться SemVer
вести CHANGELOG.md
📱 Особенности разных проектов
Библиотеки → строго SemVer
Веб-приложения → часто даты (v2023.10.15)
Мобильные → версия + build (2.0.0-build.123)
✅ Лучшие практики
Тег = только стабильный код
Не тегайте "сырые" коммиты

Используйте pre-release:

v1.0.0-beta.1
v1.0.0-rc.1
Проверяйте всё перед релизом
Автоматизируйте процесс
💡 Итог
Теги = стабильные точки в истории
Annotated теги — стандарт для релизов
SemVer делает версии понятными
Автоматизация = меньше ошибок

👉 Хорошая система тегов = предсказуемые и управляемые релизы
