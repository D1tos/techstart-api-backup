# Работа с изменениями в Git

После создания репозитория начинается основная работа — управление изменениями:  
**отслеживание → подготовка → фиксация**

---

## 🔍 Проверка состояния — `git status`

Показывает текущее состояние репозитория:

```bash
git status

📌 Что можно узнать:

текущую ветку
изменённые файлы
подготовленные изменения
новые (неотслеживаемые) файлы
Основные секции вывода:
Changes not staged — изменения не добавлены в индекс
Untracked files — новые файлы
Changes to be committed — готово к коммиту
Краткий формат:
git status -s

Пример:

 M README.md
?? config.json

📌 Обозначения:

M — файл изменён
?? — файл не отслеживается
🔎 Просмотр изменений — git diff

Показывает что именно изменилось:

git diff

📌 Обозначения:

- — удалённые строки
+ — добавленные строки
Частые варианты:
# Изменения в рабочей директории
git diff

# Изменения в индексе
git diff --staged

# Конкретный файл
git diff README.md

# Сравнение с прошлым коммитом
git diff HEAD~1
➕ Добавление изменений — git add

Переносит изменения в staging area (индекс):

# Один файл
git add README.md

# Несколько файлов
git add file1.txt file2.txt

# Всё в текущей папке
git add .

# Все изменения
git add -A

# Частичное добавление
git add -p

📌 Важно:

В индекс добавляется текущее состояние файла
Если изменить файл после git add — изменения не попадут в коммит

💾 Создание коммита — git commit

Фиксирует изменения из индекса:

# Быстрый коммит
git commit -m "Add authentication"

# Через редактор
git commit

# Все отслеживаемые изменения (без add)
git commit -am "Update config"

# Изменить последний коммит
git commit --amend
📌 Что содержит коммит:
снимок файлов
ссылку на предыдущий коммит
автора и дату
сообщение
✍️ Как писать сообщения коммитов

Структура:

Краткое описание (до 50 символов)

Подробности:
- что сделано
- почему сделано

Fixes #123

📌 Хороший коммит = понятная история проекта

🔄 Типичный workflow
# 1. Проверка состояния
git status

# 2. Просмотр изменений
git diff

# 3. Добавление нужных файлов
git add file.js

# 4. Проверка staged
git diff --staged

# 5. Коммит
git commit -m "Add feature"

# 6. Проверка
git log -1
✂️ Частичное добавление (важно)
git add -p file.js

📌 Возможности:

y — добавить
n — пропустить
s — разбить изменения
e — редактировать вручную

👉 Позволяет делать чистые и логичные коммиты

💡 Практические советы
Делай частые маленькие коммиты

Всегда проверяй:

git diff --staged
Пиши понятные сообщения (что + почему)
Используй staging для группировки изменений
🧠 Итог

Работа с изменениями строится вокруг цикла:

status → diff → add → commit
