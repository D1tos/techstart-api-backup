# Синхронизация: push, pull, fetch
## Как Git хранит информацию об удаленных ветках
3 ветки в локальном репозитории, каждый с особой ролью
### Локальные ветки
обычные рабочие ветки
полностью под контролем
### Remote-tracking ветки
локальные копии веток из удаленного репозитория
имеют специальный формат имени origin/main, origin/develop
нельзя изменить напрямую - обновляются автоматически при синхронизации
показывают состояние удаленного репозитрия на момент синхронизации
"закладки" где находятся ветки на сервере
### Удаленные ветки
Физически находятся в удаленном репозитории.
Работа только черех команды синхронизации
#### Посмотреть все типы веток
$ git branch -a
* main                    # локальная ветка (текущая)
  develop                 # локальная ветка
  remotes/origin/main     # remote-tracking ветка
  remotes/origin/develop  # remote-tracking ветка

## Git fetch - получение без изменений
Самая безопасная команда синхронизации.
Никогда не изменяет локальные ветки или рабочую директори.
только загружает новую информацию с сервера
### Что происходит при выполнении fetch
git fetch origin
    Git устанавливает соединение с удалённым репозиторием
    Запрашивает список всех веток и их текущие позиции (на какие коммиты они указывают)
    Сравнивает с локальной информацией и определяет, какие коммиты отсутствуют локально
    Загружает недостающие коммиты и все связанные объекты (файлы, деревья)
    Обновляет remote-tracking ветки, чтобы они указывали на новые позиции

#### Выполнить fetch и посмотреть, что изменилось
$ git fetch origin
remote: Counting objects: 15, done.
remote: Compressing objects: 100% (8/8), done.
remote: Total 15 (delta 5), reused 15 (delta 5)
Unpacking objects: 100% (15/15), done.
From github.com:user/repository
   a456def..b789ghi  main     -> origin/main
   c123jkl..d456mno  develop  -> origin/develop

### Почему fetch не меняет рабочую директорию
После fetch ваша локальная ветка main всё ещё указывает на старый коммит, 
а origin/main уже указывает на новый. Это позволяет вам:
    Посмотреть, что именно изменилось, перед интеграцией
    Закончить текущую работу без неожиданных изменений
    Выбрать подходящий момент для обновления

#### После fetch можно изучить изменения
$ git log main..origin/main  # что нового на сервере
$ git log origin/main..main  # что есть локально, но нет на сервере
$ git diff main origin/main  # различия в коде

## Git pull - автоматическая интеграция
не только загружает изменения, но и автоматически интегрирует их в текущую ветку
git pull = git fetch + git merge(git rebase)

### Механика работы pull
Сначала выполняется git fetch origin (получение новых коммитов)
Git определяет, какую remote-tracking ветку нужно слить (в данном случае origin/main)
Проверяет, можно ли выполнить fast-forward merge
Если да — просто передвигает указатель локальной ветки
Если нет — создаёт merge-коммит или выполняет rebase

### 3 варианта поведения pull
#### Fast-forward merge
локальная ветка не содержит уникальных коммитов
#### Merge conflict
обе ветки содержат уникальные коммиты
#### Rebase
Переписывает историю, помещая коммиты поверх полученных
##### До pull --rebase:
      C---D  (origin/main)
     /
A---B
     \
      E---F  (main)

##### После pull --rebase:
A---B---C---D  (origin/main)
             \
              E'---F'  (main)

### Когда pull может вызвать проблемы
pull изменяет текущую ветку = требует чистого рабочего состояние.
При незакоммиченных изменениях - откажется выполнять пул

## Git push - отправка на сервер
изменяет удаленный репозиторий, отправляя локальные коммиты на сервер.
### Как работает push
git push origin main:
Git определяет, какие коммиты есть локально, но отсутствуют на сервере
Проверяет, является ли операция fast-forward для удалённой ветки
Если да — отправляет коммиты и обновляет удалённую ветку
Если нет — отклоняет операцию для предотвращения потери данных
После успешного push обновляет remote-tracking ветку
### Почему push может быть отклонен
не является fast-forward операцией.
кто-то другой успел отправить коммиты раньше
### Force push и его опасности
Force push перезаписывает историю на сервере:
#### Обычный force push (опасно!)
$ git push --force origin main
#### Заменяет на сервере:
A---B---C---F---G  (старая версия)
#### На:
A---B---C---D---E  (ваша версия)
#### Коммиты F и G потеряны!
Более безопасный вариант — --force-with-lease:
$ git push --force-with-lease origin main

Эта команда проверяет, что remote-tracking ветка соответствует состоянию на сервере. Если кто-то отправил изменения после вашего последнего fetch, операция будет отклонена.

## Upstream связи и их важность
конфигурация, которая говорит, какая удаленная ветка соответсвует локальной
упрощает команды и дает больше информации о состоянии сихронизации
### Как работает
Когда установлена upstream связь:
    git pull знает, откуда брать изменения
    git push знает, куда отправлять коммиты
    git status показывает разницу с удалённой веткой
#### Установка upstream при push
$ git push -u origin feature/new-button
#### Что происходит:
##### 1. Создаётся ветка feature/new-button на сервере
##### 2. Отправляются коммиты
##### 3. Устанавливается связь:
#####    local: feature/new-button
#####    tracks: origin/feature/new-button

#### Теперь достаточно коротких команд:
$ git pull    # вместо git pull origin feature/new-button
$ git push    # вместо git push origin feature/new-button

### Информация о синхронизации
$ git status
On branch main
Your branch is ahead of 'origin/main' by 2 commits.
  (use "git push" to publish your local commits)

Эта информация возможна благодаря upstream связи. Git сравнивает вашу ветку с remote-tracking веткой и сообщает:

    ahead — у вас есть коммиты для отправки
    behind — на сервере есть новые коммиты
    diverged — есть уникальные коммиты в обеих ветках
