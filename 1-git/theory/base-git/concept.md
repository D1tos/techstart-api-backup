# Базовые понятия Git

## Репозиторий
Место, где Git хранит всю информацию о проекта. 
Физически - обычная папка, в которой создана специальная скрытая директория .git

Структура при инициализации репозитория Git

project/
├── .git/          # Служебная папка Git
│   ├── objects/   # Хранилище всех версий файлов
│   ├── refs/      # Информация о ветках
│   ├── HEAD       # Указатель текущего положения
│   └── config     # Настройки репозитория
├── src/           # Файлы проекта
├── docs/          # Файлы проекта
└── README.md      # Файлы проекта

.git содержит всю историю изменений,все версии файлов и метаинформацию.
Остальное содержимое - рабочие файлы проекта,которые можно редактировать обычным способом

### Обычный репозиторий
Рабочие файлы, история изменений.
Повседневная работа над проектом

### Голый репозиторий
Только история без рабочих файлов.
Применяется на серверах для хранения кода.
на сервере никто не редактирует напрямую

## Коммит
Зафиксированное состояние проекта в определенный момент времени.
Каждый коммит содержит:

-Полный снимок всех файлов проекта на момент создания коммита
-Уникальный идентификатор(хеш)
-Информацию об авторе и времени создания
-Описание изменений
-Ссылку на еоммит родителя

Коммиты неизменяемы. Попытка изменить = создание нового коммита

## Ветка
Именованный указазатель на определенный коммит.
это текстовый файл, содержащий идентификатор коммита

Характеристики веток:
-Создание ветки не копирует файли/историю
-При создании нового коммита ветка автоматически передвигается на него
-Можно создавать неограниченное количество веток
-Переключение между ветками изменяет содержимое рабочей папки

main | master - Стандартная ветка в новом репозитории
Доп.ветки создаются для разработки новых функций,ипраления ошибок или экспериментов

HEAD - текущее положение в репозитории/ активная ветка, указывающая на последний коммит

## Как компоненты работают вместе
Репозиторий - инфраструктура для хранения данных.
Внутри репозитория создаются коммиты
Коммиты формируют историю изменений
Ветки - именованный указатели на коммиты. Позволяют организовать параллельную разработку.

Типичный процесс

 В репозитории есть ветка main, указывающая на последний коммит
 Создаётся новая ветка feature для разработки новой функции
 В ветке feature создаются новые коммиты
 Ветка feature автоматически передвигается на новые коммиты
 Ветка main остаётся на прежнем месте
 После завершения работы изменения из feature вливаются в main
